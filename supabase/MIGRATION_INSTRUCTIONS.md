# Исправление конфликтов уникального ключа users_pkey

## Проблема

Ошибка: `duplicate key value violates users_pkey` возникает когда:
- Sequence `users_id_seq` не синхронизирован с максимальным `id` в таблице
- Попытка вставить пользователя с `id`, который уже существует

## Решение

### Шаг 1: Примените исправленную миграцию в Supabase

1. Откройте SQL Editor в Supabase Dashboard:
   ```
   https://app.supabase.com/project/<YOUR_PROJECT_ID>/sql
   ```

2. Выполните содержимое файла `schema_custom_auth_fixed.sql`

   Этот файл:
   - Устанавливает правильную структуру таблиц с `GENERATED BY DEFAULT AS IDENTITY`
   - **Отключает RLS** для development (иначе запросы будут блокироваться)
   - Синхронизирует все sequences с максимальными значениями `id`
   - Создает необходимые индексы и права доступа

### Шаг 2: Проверьте структуру таблицы users

После миграции выполните проверку:

```sql
-- Проверить структуру
SELECT 
  column_name,
  data_type,
  column_default,
  is_nullable
FROM information_schema.columns
WHERE table_name = 'users' AND table_schema = 'public'
ORDER BY ordinal_position;

-- Проверить, что sequence привязан правильно
SELECT pg_get_serial_sequence('users', 'id') as sequence_name;

-- Проверить текущее значение sequence
SELECT last_value, is_called FROM users_id_seq;

-- Проверить максимальный id в таблице
SELECT MAX(id) FROM users;
```

Ожидаемая структура:
- `id` — `bigint` — `nextval('users_id_seq'::regclass)` — `NO`
- `name` — `text` — `null` — `YES`
- `email` — `text` — `null` — `NO` (unique constraint)
- `password_hash` — `text` — `null` — `NO`
- `created_at` — `timestamptz` — `now()` — `NO`

### Шаг 3: Если есть дубликаты id (крайний случай)

Если в таблице уже есть дубликаты `id` (что маловероятно из-за PK constraint), нужно их очистить:

```sql
-- Найти дубликаты
SELECT id, COUNT(*) FROM users GROUP BY id HAVING COUNT(*) > 1;

-- Удалить дубликаты, оставив одну запись (СНАЧАЛА СДЕЛАЙТЕ БЭКАП!)
DELETE FROM users 
WHERE ctid NOT IN (
  SELECT MIN(ctid) FROM users GROUP BY id
);
```

Затем сбросить sequence:
```sql
SELECT setval('users_id_seq', COALESCE(MAX(id), 0) + 1, false) FROM users;
```

### Шаг 4: Перезапустите приложение

После выполнения миграции:
1. Остановите приложение
2. Очистите кэш (опционально)
3. Запустите заново

Ошибка должна исчезнуть.

## Архитектурные исправления

### Код Flutter (database_service_supabase.dart)
- Метод `registerUser()` НЕ передаёт `id` вручную → id генерируется автоматически PostgreSQL
- Используется `insert({...}).select().single()` → получаем сгенерированный id

### Миграция (schema_custom_auth_fixed.sql)
- `id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`
- `ALTER TABLE ... DISABLE ROW LEVEL SECURITY` — отключено для development
- `SELECT setval(...)` — синхронизирует sequence после каждой миграции

## Важно

- **Не включайте RLS** (enable row level security) пока не добавите аутентификацию через Supabase Auth
- Если планируете использовать Supabase Auth, нужно перейти на `schema.sql` и переписать логику аутентификации
- В текущей архитектуре используется **кастомная аутентификация** с хешем пароля в таблице `users`
